# StockRadar — Claude Code 워크플로우

## 작업 원칙
- 파일 수정 전 반드시 현재 내용 확인 (read-before-write)
- 한 번에 하나의 기능/버그만 처리
- 변경 파일 3개 초과 시 계획 먼저 보고 → 승인 후 진행

## 질문할 것 vs 바로 진행할 것

### 반드시 먼저 물어볼 것
- DB 스키마 변경 (마이그레이션 추가 — 현재 v8, 다음은 v9)
- 기존 API 엔드포인트 삭제 또는 시그니처 변경
- 새 npm 패키지 추가
- StockRadar_v5.html 구조적 변경
- 인증/보안 관련 로직 수정
- express-session 또는 middleware.js 변경

### 물어보지 말고 바로 진행할 것
- 명확한 버그 수정 (에러 메시지 있는 경우)
- console.log 정리
- 주석 추가 또는 개선
- 새 API 엔드포인트 추가 (기존 건드리지 않는 경우)
- .env.example 업데이트

## 보안 규칙 (항상 적용)

### API 키 / 환경변수 처리
- .env 파일의 값을 터미널 출력·로그·주석에 절대 노출 금지
- 환경변수는 반드시 변수로만 참조해서 사용 (예: K="$FMP_API_KEY")
- curl 명령어 출력 시 apikey= 이후 값은 ***로 마스킹해서 보여주기
- 작업 완료 보고에 키 값 포함 금지

### 민감 파일 취급
- .env 파일 내용 전체 출력 금지
- 키 길이·존재 여부만 확인하고 보고할 것
- 새 환경변수 추가 시 .env.example에는 키 이름만, 값은 빈칸으로

## /check 명령
"지금 /check 해줘"라고 하면 아래를 순서대로 실행:
1. server/ 내 TODO / FIXME / console.error 전체 검색
2. package.json 의존성 점검 (node-fetch 제거 여부 등)
3. .env.example에 있는 키가 코드에서 모두 사용되는지 확인
4. 미사용 라우트 또는 데드코드 탐지
5. db.js 마이그레이션 버전 일관성 확인
6. admin.js Gemini 직접 호출 잔존 여부 확인
7. 결과를 docs/audit-YYYYMMDD.md 파일로 저장
